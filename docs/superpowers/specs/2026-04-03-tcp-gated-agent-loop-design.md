# TCP-EXP-2: TCP-Gated Agent Loop Experiment

## Context

The TCP Harness Prototype validated binary descriptor filtering at 90-tool scale (MT-3) with 0 false allows, 0 false rejections, and 35,653B mean prompt reduction. All 9 prior work items passed. What remains unproven: whether the computational savings translate to measurable speedup when a real LLM processes the tool schemas.

This experiment answers a single empirical question: **does TCP pre-filtering reduce Claude first-token latency, total response time, and input token count without degrading tool-selection correctness?**

The code produced is apparatus for this question, not product.

## Architecture

Three components, all inside the existing repo:

### 1. Schema Bridge (`tcp/harness/schema_bridge.py`)

Converts TCP `ToolRecord` projections into Anthropic API `tools` parameter format.

```python
def tool_record_to_anthropic_schema(record: ToolRecord) -> dict:
    """Convert a single ToolRecord into an Anthropic tool schema dict.
    
    Returns: {"name": ..., "description": ..., "input_schema": {...}}
    """

def corpus_to_anthropic_schemas(
    entries: list[CorpusEntry],
) -> list[dict]:
    """Convert the full MT-3 corpus into Anthropic tool schemas.
    
    Generated once, then subsetted by the benchmark.
    Schema text is identical for any given tool regardless of
    which arm (filtered/unfiltered) uses it.
    """
```

**Schema parity invariant**: Both arms use schemas generated by the same function. The filtered arm selects a subset of the pre-generated list. No re-generation, no reformatting. The only variable is tool count.

**EXP-1 preservation**: Tools with `AUTH_REQUIRED` in capability_flags get a description annotation indicating approval-required status. This maintains the three-tier distinction in the model-visible schema.

### 2. Agent Loop (`tcp/agent/loop.py`)

A minimal instrumented agent loop using the Anthropic SDK.

```python
@dataclass(frozen=True)
class LoopMetrics:
    task_name: str
    tool_count: int
    turns: int
    first_token_latency_ms: float
    total_response_time_ms: float
    input_tokens: int
    output_tokens: int
    tools_called: tuple[str, ...]
    selected_tool_correct: bool
    error: str | None

async def run_agent_loop(
    task_prompt: str,
    tools: list[dict],
    mock_executor: Callable[[str, dict], str],
    *,
    model: str = "claude-sonnet-4-6",
    max_turns: int = 5,
) -> LoopMetrics:
    """Execute a single agent loop and return metrics.
    
    1. Call messages.create() with task_prompt and tools
    2. If response contains tool_use, dispatch to mock_executor
    3. Feed tool_result back, repeat
    4. Collect timing at every API call boundary
    """
```

**Mock executors**: Each tool in the corpus has a canned response registered in a lookup table. The mock returns valid-shaped JSON for the tool's declared output format. This isolates the measurement to model-side behavior.

**Instrumentation points**:
- `time.perf_counter_ns()` around each `messages.create()` call
- Token counts from the API response `usage` field
- Use non-streaming calls; first-token latency = total wall-clock for the first API call in the loop (before any tool execution round-trips)
- Tool name extracted from `tool_use` content blocks

### 3. Benchmark Runner (`tcp/agent/benchmark.py`)

Runs paired trials and produces comparison reports.

```python
@dataclass(frozen=True)
class PairedTrial:
    task_name: str
    unfiltered: LoopMetrics
    filtered: LoopMetrics
    
    @property
    def latency_delta_ms(self) -> float:
        return self.unfiltered.total_response_time_ms - self.filtered.total_response_time_ms

    @property
    def token_delta(self) -> int:
        return self.unfiltered.input_tokens - self.filtered.input_tokens

@dataclass(frozen=True)
class BenchmarkReport:
    trials: tuple[PairedTrial, ...]
    summary: dict[str, float | int]

async def run_paired_benchmark(
    tasks: list[BenchmarkTask],
    corpus_schemas: list[dict],
    filtered_schemas_by_task: dict[str, list[dict]],
    *,
    repetitions: int = 5,
    model: str = "claude-sonnet-4-6",
) -> BenchmarkReport:
    """Run paired filtered/unfiltered trials for each task.
    
    Order is randomized per pair to control for prompt caching.
    """
```

**Paired design**: For each task, run unfiltered and filtered back-to-back in randomized order. This controls for network conditions and prompt-cache state. Repeat N times per task. Report per-pair deltas and aggregate statistics.

**Task fixtures**: Reuse the 12 MT-3 tasks. Each task has:
- A natural language prompt (e.g., "Read the file at /tmp/data.json and extract the 'name' field")
- An expected tool (e.g., `fs-read-file` then `jq`)
- The TCP-filtered tool subset for that task (from `bitmask_filter`)

## Metrics

| Metric | How measured | Success threshold |
|--------|-------------|-------------------|
| Input token reduction | `unfiltered.input_tokens - filtered.input_tokens` | > 0 for all tasks |
| First-token latency delta | Paired wall-clock difference | Filtered faster on aggregate (p < 0.05 or consistent sign) |
| Total response time delta | Paired wall-clock difference | Filtered faster on aggregate |
| Tool selection correctness | Expected tool == chosen tool | Filtered >= unfiltered (no degradation) |
| Schema bridge coverage | ToolRecords that convert to valid Anthropic schemas | >= 80% of corpus (72+ of 90) |

## Kill Conditions

Stop immediately if:

1. **False allows in filtered arm**: TCP admits a tool that shouldn't be available for a task
2. **Correctness regression**: filtered arm selects the wrong tool more often than unfiltered
3. **Variance dominates signal**: after all repetitions, API jitter is larger than the observed latency delta (effect indistinguishable from noise)
4. **Schema bridge coverage collapse**: < 72 of 90 ToolRecords convert to valid Anthropic schemas
5. **Loop-induced invalidity**: > 20% of failures are loop bugs (malformed mock responses, SDK errors) rather than model behavior
6. **Cost blowout**: filtered runs show no latency or correctness benefit while significantly increasing experimental complexity

## Dependencies

- `anthropic` Python SDK (new dependency, added to pyproject.toml dev group)
- `ANTHROPIC_API_KEY` environment variable
- Existing: `tcp.harness.corpus`, `tcp.harness.bitmask_filter`, `tcp.harness.models`

## Model Choice

Use `claude-sonnet-4-6` for benchmarking. It's fast enough for multiple repetitions to be affordable, and representative of real agent workloads. The experiment should parameterize the model so it can be re-run on other models later.

## What This Does NOT Do

- Does not build a production agent system
- Does not modify Claude Code internals
- Does not run real tool executions (all mocked)
- Does not claim end-to-end agent productivity gains — only model-side selection and latency behavior under different context loads
